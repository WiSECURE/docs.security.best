A. Business Overview

A.1 The Emergence of Google CSE

To provide users with trust in using Google Workspace, Google designed the GWS Client-Side-Encryption (CSE) option for GWS enterprise customers. Customers concern about Google's security can turn on the CSE option for their GWS subscription.

The basic concept of CSE is to provide a customer-controlled or trusted third-party-controlled critical management service, CSE Key Access Control List Service (KACLS), to manage the encryption keys (AES keys) used to encrypt GWS service items. GWS defines a set of KACLS APIs to co-work with the browser-side JavaScript. These API functions will replace the original key management functions in the browser-side codes. In addition to the encryption keys, GWS Gmail requires public-key operations. These APIs also support public key cryptography.

CSE was launched in November 2023, collaborating with Thales (https://thalesdocs.com/ctp/cm/2.8/admin/cckm_ag/gws/index.html). The initial CSE option is to provide the SaaS-based CSE key management service on the cloud, which leverages the cloud HSM:

The core security of Thales KACLS is mainly based on SafeNet Trusted Access and Thales Cloud HSM. SafeNet Trusted Access provides organizations in GWS with identity authentication and optional provider services (STA IdP). After user identification and authentication, an authorization token is created for the user session. Then Thales KACLS uses the token to authorize key services, which help GWS applications in the browser to protect their data keys or decryption/signing private keys. The cloud HSM services perform all key protection operations.

A.2 WiSECURE KACLS Initiative

Although all the services from GWS are cloud- or SaaS-based, which makes the deployment and configuration easy to manage, the major problem for this deployment model is still the "trust." Users may still not trust Google, even if they claim all the users' files or streaming videos are encrypted and never disclosed by other parties or by Google itself. In the same thinking, there is no way to persuade customers to trust cloud-based HSM services and the access control based on the cloud-based IdP neither. The key management service is more sensitive than simple IdP and access management. That's because once the keys are compromised, the only solution to mitigate the security risk is to revoke them and reissue them. All the previously encrypted data is no longer safe, and there is no resolution to retract the disclosed data.

Due to security concerns about cloud-based key management services, on-premise key management is a better choice for CSE KACLS. It's an initiative of the WiSECURE KACLS solution. WiSECURE GWS CSE aims to ensure "security under your eyes and within your fingertips."

WiSECURE provides the following components for GWS CSE integration:

  1. Security Token: provide the FIDO2 and key wrapping, unwrapping, and signing capabilities

  2. WiSECURE KACLS Agent: the local GWS CSE service which communicates with the security token

  3. WiSECURE Key Management & Distribution Center (KMDC): for the security token issuance and management

  4. WiSECURE KACLS Control Server (CS): to provide the audit and revocation management for the security token

  5. WiSECURE KACLS Log Service: to keep or forward all activity logs from the security tokens of the user side.

The components collaborate, as shown in the following diagram:

With the WiSECURE KACLS, GWS CSE users can leverage FIDO2-based password-less identity authentication and reliable hardware-based key management in GWS.

Since Google Workspace JavaScript codes running in the browser handle the keys locally, no cloud service providers know the key materials except the browsers themselves. All data key protection is performed by the hardware with the high security level.

There are still two limitations of WiSECURE solutions regarding the quality of the complete cryptographic services:

  1. The data is encrypted by standard AES cryptography. Unless Google has an alternate implementation (e.g., creating a browser plug-in API to invoke another software/hardware entity), the symmetric algorithm cannot be customized according to the CSE customer's security policy.

  2. The software codes within the browser perform the data key generation, which may not have the hardware-level quality in the keying material.

Except for the above limitations, a Security Token can help the browser hand over the data keys used in encryption/decryption/signing/envelope-decryption to trustworthy hardware.

WiSECURE GWS CSE KACLS has a Master Key (MK) stored inside the Security Token to control the root of trust in security. The key-encryption key (KEK) is derived from the master key and entropy. All data keys are wrapped by KEK inside the security token before they are uploaded to Google GWS with encrypted contents. When other users than the content creator need to access the content, their own Security Tokens are used to unwrap the data keys.

For those services requiring public key cryptography, WiSECURE KACLS works similarly. Security Tokens will protect private keys (e.g., the signing key of S/MIME in Gmail). When the browser needs to sign the messages, it only sends the hash value of the messages and the private key index to the Security Token. The hardware will sign for the browser after the successful user PIN verification. To decrypt the public-key-sealed S/MIME envelope works the same way. The data key used to encrypt the envelope will be sent to the security token to unwrap by the user's private key.

Security Token works with WiSECURE KACLS Agent. The agent is only responsible for receiving the KACLS API requests from the browser JavaScript and responding to the browser according to the token cryptographic operation results. All the critical cryptographic operations are performed securely inside the tokens.

Since the Security Tokens are distributed from the administrative authority to the end users, there are two security management considerations to be ensured:

  1. how to control the keys and trusted information reliably delivered into Security Token,

  2. how to precisely ensure the dongle owners are the actual users

To control the keys inside the tokens, WiSECURE KACLS provides WiSECURE Key Management & Distribution Center (KMDC) software to coordinate:

  • the key generation,

  • keying material,

  • user account information injection, and

  • the token control keys.

KMDC helps the GWS administrators to associate any user account with their token. It also generates the control keys for the status management of Security Tokens. In addition to the MK, the trusted information, such as the URL and certificates of control services and the agent, will also be written into the secure storage inside the tokens. The agent will follow the information obtained from the tokens to respond to the KACLS API requests.

To guarantee no user other than the token owner can use WiSECURE KACLS services, there is a PIN (Personal Identification Number) to ensure the ownership of the token. There is a configurable PIN retry counter for every security token. Once the number of retrials exceeds the counter, the token will lock itself and wait for the authorized people to unlock or reset it. In addition to PIN protection, the token control key can be revoked anytime when the administrators of KMDC think there are any potential risks about the events of improper usage or loss of the hardware.

To enforce the above controls, WiSECURE KACLS provides Control Server (CS) software and Log Service components. The administrators can decide whether to deploy these software on-premise or cloud-based. The agent will always consult the validity of the dongle for each KACLS API request with CS and push the logs of all operations of Security Tokens to the Log Server for logging events.

The Log Server is responsible for keeping or forwarding all the events to other log collectors. To guarantee their integrity and authenticity, all the events are cryptographically signed by Security Tokens by the control keys.

B. GWS and WiSECURE KACLS Integration

B.1 WiSECURE USB Token

The major difference between SaaS KACLS and the WiSECURE solution is that we move the key management services to the client side. This approach reduces the efforts of access control, but it relies on the security strength and good management of the local device (token).

Only the owner can access the key management services. There are no complex authentication and authorization mechanisms for this access.

However, if the token is easy to commit fraud or the keys inside the token are easy to compromise, it will be a disaster for the organizations.

To ensure the high security level, WiSECURE KACLS solution uses the Common Criteria (CC)-certified hardware to protect the keys and all the cryptographic operations. The solution generates the keys inside the device or uses secure issuance software to inject them into the token.

WiSECURE KACLS provides two types of tokens:

  • commercial-grade USB HSM with CC EAL5+ security element inside (WiSECURE Blentity) https://wisecure-tech.jp/wp-content/uploads/2024/01/Blentity-user-manual.pdf

  • security-enhanced USB HSM (kvKey Gold, under FIPS 140-3 Level 3 certification)

Blentity and kvKey Gold tokens can work together, depending on the security level requirements of the CSE organization. The difference between them is the cryptographic service interfaces. kvKey Gold provides a PIV-compliant programming interface, which allows public key cryptography to be performed by other software entities.

Only Blentity is supported in the current version of WiSECURE KACLS. After kvKey Gold obtains the FIP 140-3 certification, it will be adapted into the solution.

WiSECURE Blentity token is also a FIDO authenticator. It supports WebAuthn CTAP 2.1 protocols. The user can turn on GWS MFA security option and use the USB token to log-on Google account.

For information on the security management of USB tokens, please refer to "B.4 The Security Management" and "5. Security Reference."

B.2 The Usage of GWS CSE with WiSECURE KACLS

B.2.1 Using Gmail in WiSECURE KACLS

The management of public keys and certificates in CSE differs from other GWS CSE services. Gmail follows standard S/MIME format to provide message body signing and content protection.

The signing and encryption processes handled by Gmail require the following sensitive cryptographic services:

  1. For signing the message, the signer's private key is needed;

  2. In sending encrypted emails to other recipients, their private keys are required to decrypt the envelope data keys.

WiSECURE KACLS Agent receives the requests for the above two services and forwards all essential private key operations to the USB tokens.

In preparing Gmail encrypted S/MIME contents, the GWS JavaScript also uses the registered certificates to wrap the envelope data keys. The wrapping process doesn't need the help from KACLS. Gmail will get the certificates from its repository. WiSECURE KMDC software will upload the certificates automatically when the tokens are issued.

For PIV USB tokens, the administrator can prepare their private keys and certificates with other software (e.g., Microsoft ADCS). In WiSECURE KMDC software, the administrators must associate the private key label for Gmail S/MIME usage before the token issuance is completed.

WiSECURE KACLS Agent supports the API for the private key services on the user's token. The keying material is never disclosed to the Agent software. The key label is used to identify the user's private keys.

B.2.2 Using Calendar/Docs/Meet in WiSECURE KACLS

CSE features for GWS Calendar, Google Docs, and Google Meet are simpler than those for Gmail.

The key management functions for these services are only focused on protecting DEKs. The WiSECURE KACLS solution wraps each DEK by the KEKs.

When the browser creates the service contents, the DEK is generated inside the browser. Then the browser will pass the user-account information and the keying material of DEK to the Agent. The browserâ€™s JavaScript passes the parameters by KACLS API service requests to the local port of the Agent. The agent will transparently forward this information to the USB token. Finally, the Agent returns the wrapped DEK by the content ownerâ€™s KEK. On the decryption side, since the service contents are encrypted by the userâ€™s DEK and get shared with receiving parties, the Agent will pass the service content owner's user account information to the token. The token will use it instead of the token ownerâ€™s account information to derive the associated KEK and unwrap the wrapped DEK. With the help of the Agent and the USB token, the browser will obtain the original DEK of the service contents.

For the WiSECURE KACLS solution, every USB token can decrypt the DEK, which is wrapped by any user (within the same organization) account. This results in some management vulnerability for such distributed KACLS mechanisms. To mitigate the potential risk, the WiSECURE KACLS solution provides users with strictly controlled security management. In addition to the USB token at the high-security level, some security controls are enforced to reduce the impacts of token loss incidents and MITM attacks.

B.2.3 The WiSECURE KACLS Agent

WiSECURE KACLS Agent runs inside a CSE client-side OS, such as Windows or macOS. It listens to the local network port and adapts KACLS API requests to secure tokens.

The major function of the Agent is to establish secure communication between the browser and itself. After receiving API requests, it extracts all sensitive operations to tokens.

In addition to the service bridge, it also performs the following auxiliary security assistant tasks:

  • the token PIN log-in to guarantee the ownership,

  • forward the user session credential to the token for DEK wrapping/unwrapping,

  • enforce the configuration information integrity provided by the token,

  • validate the token by the communication between Control Services and itself,

  • prompt activity event logs signing requests to the token,

  • send the signed logs to the Log Service, and

  • change the token PIN setting

Although the Agent is never involved in sensitive data processing and key management, it plays a critical role between the browser and the token. Some potential MITM attacks are possible. Please reference "5. Security Reference.â€

B.3 The Token Issuances

To protect keys, user account information, and critical configurations, the WiSECURE KACLS solution provides secure provisioning tools for token issuance. The major goal of the tool is key management and distribution. The solution is called WiSECURE KACLS Key Management and Distribution Center (KMDC).

KMDC performs the following administrative tasks:

  1. to generate device key pairs for the token management,

  2. to sign the device certificate to ensure the token validity,

  3. to provision the user account information of the token owner,

  4. to securely inject MK into the token,

  5. to securely inject the critical configurations into the token, which the Agent will reference,

  6. to generate the Gmail key pairs and Certification Service Request (CSR) for the user account,

  7. to associate the Gmail certificate with the token, and

  8. to publish all the user information and certificates to the Google repository

The Gmail certificate is not necessarily signed by WiSECURE KMDC. The administrators can export the CSR material to any public CAs to obtain the Gmail user certificates. This implies the certificates can be acquired after the token issuance process. KMDC will keep track of the certification status for any specific token.

In order to control the validity of the token, once the administrator decides to revoke any token, KMDC will update the validity state of the specific token to WiSECURE KACLS CS.

Because KMDC will upload the account associations to the Google repository, the Google-Workspace-domain-delegated service account credentials are required. To ensure security, this sensitive information, KACLS MK, and other organization-related configurations have to be protected by the hardware token, the administratorâ€™s token.

When WiSECURE KMDC works, it requires the administrative token to log on to Google administrative services through the API and retrieves all sensitive materials from the administrative token.

In the future release of KMDC, the PIV support will be included.

B.4 The Security Management

To ensure the USB tokens are used appropriately, the WiSECURE KACLS solution will have the following security designs:

  1. The revocation of problematic tokens

  2. The proven token ownership

  3. The Agentâ€™s integrity protection

  4. Activity logs

Using the USB token by its exact owner is the only way to guarantee access permission for KACLS key management services. Items #1 and #2 of the above security designs are for this.

When it's issued, some token-specific key pairs are created. WiSECURE KMDC software signs its public keys as the token certificates. Once the token has potential security concerns, the administrators revoke the certificates and publish their revocation status to the Control Services.

Every time a token is used, the Agent checks its validity. If it is revoked, the Agent rejects the use of the token immediately.

The Agent will detect the plugged/unplugged events continuously. Once a newly plugged token is found, the log-in process will be performed using PIN verification. For more security control, the token user can enable Google account MFA to ensure the association between the account and the hardware device.

Many sensitive configurations are securely saved inside the token. Without the issuance authority, nobody can write the fraud configurations into the token hardware. The Agent has no software-based configuration stored in the client-side OS. All behaviors are determined by the configurations obtained from the token.

The Agent keeps every activity event for each token operation. These events are signed by the tokenâ€™s private key and sent to the Log Server for incident analysis. The WiSECURE KACLS Log Service can be configured to forward the collected logs to another threat detection engine. Once there is any potential risk associated with the specific token, the administrators can revoke it, which disables the token in the next KACLS service request.

C. The Deployment of WiSECURE KACLS

There are two options for WiSECURE KACLS deployment:

  1. The standalone USB token deployment

  2. On-promises KACLS server deployment

C.1 USB Standalone

The standalone USB token deployment comes with the following:

  1. Security tokens for each user,

  2. Agent software package for user Windows/macOS installation or read-only partition in the Blentity storage,

  3. KMDC software with specific security tokens for the administrators,

  4. container-based Control Server software,

  5. container-based Log Service software.

Due to the network configuration restriction, the administrators must consider how to deploy Control Services and Log Server. The WiSECURE KACLS solution will not work without the proper configuration due to the validity checks and logged event forwarding problems.

C.2 On-premises KACLS Server

On-promises KACLS server deployment replaces the standalone local Agent software and the hardware tokens with the KACLS server machine under the control of the KACLS organization.

That is, the functions in items #1 and #2 of the standalone USB deployment will be moved to the WiSECURE KACLS server box. To ensure the server's cryptographic operation is secure enough, the PCIe-based HSM product kvHSM from WiSECURE will be used.

WiSECURE kvHSM HSM is a FIPS 140-2 Level 3 certified product. Itâ€™s secure hardware can provide AES, RSA, and ECDSA/ECDH services.
