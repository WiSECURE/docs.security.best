B. Prerequisites

During the initial setup process, two roles are required:

  1. Google Workspace Super Admin (Administrator Seed Role)

  2. The administrator of the Log Service and the Control Server

The same individual may perform these roles.

You will need a Google Workspace Super Admin account to configure settings in the
Google Workspace Admin Console and Google Cloud.

Please refer to the detailed instructions in Upload Encryption Keys for CSE to complete the setup process.

B.1. Establish KACLS URL

During the initial setup, the Google Workspace Administrator will use this URL to communicate between Google Workspace and the KACLS service. Also, in this step, the certificate and keys of WiSECURE KACLS Agent need to be prepared. These will be used in Configure External Key Service.

The steps may vary depending on your domain provider, but generally, the steps are as follows:

  1. Choose the domain you want the WiSECURE KACLS Agent to use.

  2. Point this domain to 127.0.0.1 for A record type and/or ::1 for AAAA record type (IPv6 only).

  3. The KACLS URL will be formatted as follows: https://<domain-chosen-above>:8443/.

In this document, https://kacls.py.security.best:8443 is used as an example.

B.2. Configure External Key Service

  • In the Admin Console, navigate to Data → Compliance → Client Encryption, and configure the External Key Service, using the IdP (Identity Provider) (this document uses Google's IdP).

  • Click "Add", and specify Name and enter URL created in Establish KACLS URL.

  • Launch the WiSECURE KACLS Agent in demo mode. This will require you to provide a valid HTTPS certificate/private key pair to listen to the 8443 port on your computer. In this mode, the KACLS agent will only respond to Google’s test connection function and won’t be able to encrypt/decrypt any data.

  • Click TEST CONNECTION to test that Google Workspace can communicate with the KACLS agent.

    If the connection fails, correct the URL and retry.

  • Click ADD SERVICE. The Google Workspace connection to KACLS is configured.

B.5. Configure CSE S/MIME Certificates for Users

  1. In the Admin Console, navigate to Apps → Google Workspace → Gmail.

  2. Click on User Settings in the right-hand column.

  3. Select S/MIME and enable the following options

    • Allow S/MIME encryption for sending and receiving emails.

    • Allow users to upload their own certificates.

  4. (If using self-signed S/MIME certificates) Under "Accept additional root certificates for specific domains," click Add.

  5. (If using self-signed S/MIME certificates) Upload the root certificate and enter the domain.

B.9. Summary

Before using the KMDC, you will need to prepare:

  • A TLS certificate for the WiSECURE KACLS Agent, which is signed for the domain of the KACLS URL

  • TLS certificate for the WiSECURE Log Service and WiSECURE Control Server. Depending on your configuration, you can use a reverse proxy to archive the goal of putting these services under the same domain.

  • (For self-signed S/MIME certificates only) A S/MIME bundled CA certificate includes a root CA certificate and at least one intermediate CA certificate.

D. Operate Tokens

To enable users to perform secure operations with apps in Google Workspace, you will need to issue or revoke tokens as necessary

D.1. Issue a token

Configure TLS and S/MIME Certificates, and Write User Information into Tokens

  1. Configure TLS and S/MIME Certificates:

    • Ensure that TLS and S/MIME certificates are properly configured and installed as required for the KACLS services. This includes setting up the certificates on the KACLS server, WiSECURE Log Service, and WiSECURE Control Server.

  2. Write Information into Tokens:

    • Incorporate the configured TLS and S/MIME certificates, along with user information, into the tokens. This will enable users to securely perform operations within the KACLS service.

By following these steps, users will be able to interact with WiSECURE KACLS Agent securely and efficiently.

D.2. Revoke a token

Revoke Certificate for a Specific Token

To prevent a user from performing any operations within WiSECURE KACLS Agent, you need to revoke the certificate associated with a specific token by issuing a revocation request to the control server.

  1. Identify the Token:

    • Determine the specific token for which you need to revoke the associated certificate.

  2. Send Revocation Request:

    • Issue a revocation request to the WiSECURE Control Server to revoke the certificate associated with the identified token.

  3. Verify Revocation:

    • Confirm that the revocation request has been processed successfully and that the user can no longer perform any operations within the WiSECURE KACLS Agent.

By completing these steps, you ensure that the specified token is invalidated and the user’s access to WiSECURE KACLS Agent is effectively blocked.

F. Q&A

  1. How to bind the Google Workspace account with the hardware token?

    Refer to Issue-Token “Select the Google Workspace Account to Bind with”

  2. Why I need to associate the user certificates with the user’s token?

    Specify User Certificate Information, this allows me to manage the correspondence between personnel and their certificates.

  3. How does the Key Management and Distribution Center software help the users to bind their account information?

    Upload User Certificate Information to Google, this allows me to establish the correspondence between personnel and their certificates on Gmail.

  4. Can I postpone S/MIME certificate signing after the token issuance?

    Delay the Configuration/Upload of User Certificates, this allows me to save time by postponing the certificate configuration or upload until the certificates are issued.

  5. Can I erase any issued token and re-issue it for the same user or another user?

    Revoke any token, this allows me to prepare for reissuing the token.

  6. Can I revoke any issued token to invalidate its right to use in WiSECURE CSE?

    Review the Key Management and Distribution Center main page. On the Token List tab, you can revoke any token that is issued. Refer to Revoke Token.

  7. Can I extend the validity of the tokens?

    Update the Certificate Information for an Issued Token, this allows me to extend the token's validity period.

  8. How can I check the logs about the token issuance?

    Review the Key Management and Distribution Center main page, the record of every issued token will be on the "Token List" tab.

  9. How can I know the token user’s activities in Google Workspace?

    In Google Admin console, Reporting → User Reports can see user’s activities.

  10. Can I back up all the critical material within the token for disaster recovery?

    Backup and Restore All Related Certificate Information, this ensures that I can return to the backup state in the event of a disaster recovery.
