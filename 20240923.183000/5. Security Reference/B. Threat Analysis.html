B. Threat Analysis

B.1. Assets

The assets considered in the WiSECURE CSE Solution are:

  • Customer data in Google Workspace (e.g., docs, sheets, slides), for which confidentiality shall be protected.

  • High availability is required for the solution.

B.2. Threats

B.2.a. Malware

Hardware-based encryption devices operate solely on secure hardware, which results in no standard operating systems. They are mainly based on hardware circuits and lack software components, making protecting against malware and virus attacks easy.

Due to its hybrid solution, not only pure software, the reverse engineering is more complex.

B.2.b. DDoS

A denial-of-service attack (DDoS attack) is a cyber-attack in which the perpetrator seeks to make a machine or network resource unavailable to its intended users by temporarily or indefinitely disrupting the services of a host connected to a network.

Distributed denial-of-service attack (DDoS attack), the incoming traffic flooding the victim originates from many sources.

Defense in depth describes the multiple layers of defense that protect Google’s network from external attacks. It starts with industry-standard firewalls and access control lists (ACLs) to enforce network segregation. All traffic is routed through custom Google Front End (GFE) servers to detect and stop malicious requests and Distributed Denial of Service (DDoS) attacks.

Refer to “Google Workspace security whitepaper.”

B.2.c. Supply Chain Risk Management (SCRM)

Our supply chain complies with several international standards, such as ISO 27001 and 28000 (Supply Chain Security Management), and adherence to military CMMC regulations ensures our high-standard SCRM.

B.2.d. Side Channel Attack

Our EAL 5+ HSMs are meticulously designed with robust security features to safeguard against physical and logical attacks. These features include:

  • Tamper detection and response: Automatic zeroization of sensitive data upon detecting tampering.

  • Protection against timing, power analysis, and electromagnetic attacks: Integration of countermeasures such as blinding techniques, randomization, or power equalization.

B.2.e. Crypto Analysis

Unlike other solutions, our solution implements critical cryptographic operations on the Security Token.

The keys for wrapping, unwrapping, and signing are stored and protected by the CC EAL 5+ secure chip on the Security Token. The solution allows you to use customized crypto algorithms, including post-quantum cryptography. The Security Token has CNSA 2.0 capability.

B.2.f. Man-in-the-Middle (MITM) Attack

A man-in-the-middle (MITM) attack is a general term for when a perpetrator positions himself in a conversation between a user and an application—either to eavesdrop or to impersonate one of the parties—making it appear like a regular exchange of information is underway.

An attack aims to steal personal information like login credentials, account details, and credit card numbers. It typically targets users of financial applications, SaaS businesses, e-commerce sites, and other websites that require logging in.

Vulnerabilities to MITM attacks in WiSECURE KACLS architecture on client-side including:

  • KACLS Agent to the Security Token

  • GWS frontend to the KACLS Agent

  • KACLS Agent to the control server and the log service

All of the above are considered, and the solution uses security countermeasures in design and implementation to mitigate the risks.
